1. First Principles: The "Why" of the PioneerRx API
The PioneerRx API is designed to bridge the gap between external healthcare platforms (like Telehealth or EHR systems) and the PioneerRx Pharmacy Management System. Its core purpose is to automate the prescription lifecycle—from intake (New EScript) to modification (Edit/Refill) and financial verification (Price Check/Claims).

By using a stateless RESTful architecture secured by cryptographic signatures, it ensures that sensitive PHI (Protected Health Information) is transmitted securely while maintaining strict synchronization between the prescriber and the pharmacy's local database.

2. Technical Mechanics: Authentication & Security
Your implementation of the SHA-512 signature-based authentication is correct according to the documentation provided.

Headers Required:
prx-api-key: The unique identifier provided during enrollment.
prx-timestamp: ISO 8601 UTC format.
prx-signature: The hashed security token.
Signature Generation Logic:
Capture the ISO 8601 UTC timestamp.
Concatenate: timestamp + shared secret.
Compute a SHA-512 hash of the UTF-16-LE encoded byte array of that string.
Convert the resulting byte array to a Base64 string.
Verification: This method prevents "replay attacks" because the pharmacy server validates that the timestamp is current (usually within a 5-minute window) and that the signature matches the secret key held on their end.
3. API Endpoints & Request/Response Formats
Based on the manual, here are the specific paths and structures you requested:

A. EScript Submission (New Prescriptions)

Endpoint: POST /api/v1/Prescription/Process/NewEScript
Payload Format: JSON object containing an rx array.
Key Fields: rxID (GUID), directionsLiteral (String), prescriberID (GUID), prescribedItemID (GUID), prescribedQuantityUnit (String), and startDate (DateTime).
Response: Returns an object containing a TransactionID (GUID), RxNumber (Integer), and primaryClaimResponse.
B. Status Checking (Querying Order Status)

Endpoint: GET /api/v1/Claim/RxTransaction?rxTransactionID={id}
Function: This allows you to pull the status of a specific transaction, including claim responses and payment method details.
Secondary Check: GET /api/v1/Prescription/RxPostEdit/Active?rxTransactionID={id} checks if there are pending edits or blocks on a specific Rx.
C. Connection Test / Health Check

Unauthenticated Health Check: GET /api/v1/Test/IsAvailable (Returns true if the service is up).
Authenticated Health Check: GET /api/v1/Test/IsAvailableWithAuth (Verifies if your prx-signature logic is correctly implemented).
4. Technical Deep-Dive: Payload & Response Structures
The API uses a highly structured JSON format. Note that optional fields are marked with a "?" in the documentation.

Request Payload (Example: New EScript):
{
  "rx": [{
    "rxID": "<guid>",
    "directionsLiteral": "Take 1 tablet daily",
    "prescribedItemID": "<guid>",
    "prescribedQuantity": 30.0
  }]
}
Response Payload (Example: Refill Success): The response is verbose, providing a message (usually null on success), a RxTransactionID, and a primaryClaimResponse which includes the claimID and transmissionID. Crucial Detail: The RxNumber returned is the internal pharmacy reference number.
5. Edge Cases: Risks and Unusual Scenarios
Signature Mismatch: The most common failure point is the encoding. PioneerRx specifies UTF-16-LE. If your library defaults to UTF-8, the SHA-512 hash will not match their server's calculation.
Blocking Edits: The RxPostEdit - Unblock endpoint suggests that certain edits (like changing a quantity or drug) can trigger a "block" in the pharmacy workflow that requires an explicit API call to release.
Re-adjudication: When using POST /api/v1/Prescription/Edit/RxTransaction, the readjudicate boolean flag is critical. Setting this to true will trigger a new insurance claim, which may change the patient's copay.
6. Missing Details: Sandbox, Webhooks, and Status IDs
Based strictly on the provided images, some information is not explicitly listed and typically requires a direct data sheet from the PioneerRx integration team:

Sandbox URL: The manual uses https://pioneerserver:42957/ as a placeholder. In a production/test environment, this is usually a specific IP or a dynamic DNS address assigned to the specific pharmacy location's server.
Test Credentials: These are never printed in general manuals for security reasons. You must request a Developer Test Key and Shared Secret from your PioneerRx account manager.
Webhook Setup: The images provided do not show a Webhook/Callback configuration. PioneerRx often relies on "Polling" (using the GET Claims endpoint) rather than "Push" notifications, unless a custom integration layer is active.
Status Values: The manual mentions RxStatusTypeID: 4 - Discontinue. A full list (e.g., 1-Active, 2-Filled, 3-Voided) is usually found in an Appendix not pictured here.
Store ID: This is implicitly linked to your prx-api-key. Each API key is typically hard-coded to a specific Store/Location ID in the PioneerRx backend.
7. Actionable Steps: What to do next
Verify Encoding: Ensure your signature generation script is explicitly using UTF-16 Little Endian before hashing.
Run the Auth Test: Call the GET /api/v1/Test/IsAvailableWithAuth endpoint first. If this returns true, your authentication logic is 100% correct.
Price Check First: Before submitting a NewEScript, use the GET /api/v1/Item/Pricing/PriceCheck endpoint. This validates that the prescribedItemID you are using is recognized by the pharmacy's inventory.
Request the Status Appendix: Ask your PioneerRx contact for the full enumeration list for RxStatusTypeID and PostEditTypeID to properly handle response logic.
