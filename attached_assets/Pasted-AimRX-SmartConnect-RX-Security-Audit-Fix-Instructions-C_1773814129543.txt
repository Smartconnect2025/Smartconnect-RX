AimRX / SmartConnect-RX Security Audit Fix Instructions
CONTEXT — READ THIS FIRST
This document contains every security fix applied to the AimRX codebase after a comprehensive 7-round architect security audit. The SmartConnect-RX Replit has a copy of this same codebase and needs the identical fixes applied.

What happened
The original codebase used Supabase Row Level Security (RLS) as the primary access control mechanism. Many API routes used the browser-level Supabase client (anon key), which meant RLS policies controlled who could read/write data.

When we fixed a bug where providers couldn't create prescriptions (RLS error on provider_patient_mappings), we switched 12+ API routes from the browser client to createAdminClient() — a server-side client using the Supabase service role key, which bypasses ALL RLS policies.

This fix was necessary, but it exposed a critical gap: many routes had NO code-level access control checks — they were relying entirely on RLS. Once RLS was bypassed, any authenticated user could access any patient's data.

The fix pattern
Every route that uses createAdminClient() now has explicit code-level security:

Authentication check — verify user is logged in
Role check — verify user is a provider or admin
Provider-patient mapping check — verify the provider is mapped to the specific patient they're trying to access
Fail-closed — if provider record is missing → 403 (not silently bypass)
Key database relationships
providers.user_id → references auth.users.id (the Supabase auth user)
provider_patient_mappings.provider_id → references providers.id (NOT auth.users.id)
To check if a provider has access to a patient: find the provider by user_id, then check provider_patient_mappings using provider.id + patient.id
PREREQUISITE: Verify createAdminClient exists
File: core/database/client.ts

This file should export a createAdminClient() function that creates a Supabase client with the service role key. If it already exists, no changes needed. It should look like this:

import { createClient } from "@supabase/supabase-js";
import { envConfig } from "@core/config";
export function createAdminClient() {
  const supabaseUrl = envConfig.NEXT_PUBLIC_SUPABASE_URL;
  const serviceKey = envConfig.SUPABASE_SERVICE_ROLE_KEY;
  if (!supabaseUrl) {
    throw new Error("NEXT_PUBLIC_SUPABASE_URL is required");
  }
  return createClient(supabaseUrl, serviceKey, {
    auth: {
      autoRefreshToken: false,
      persistSession: false,
    },
  });
}
FIX 1: app/api/prescriptions/submit/route.ts
What was wrong
No role check — any authenticated user could submit prescriptions
No provider impersonation prevention — a provider could submit as another provider
No patient existence validation
No provider-patient mapping check
What to add (after the getUser() authentication check)
Add these checks IN ORDER after verifying the user is authenticated:

1a. Role check (after auth check):

if (!userRole || (userRole !== "provider" && userRole !== "admin")) {
  return NextResponse.json(
    { success: false, error: "Only providers and admins can submit prescriptions" },
    { status: 403 },
  );
}
1b. Provider impersonation prevention (after parsing request body):

// For providers, force prescriber_id to their own user.id to prevent impersonation
// For admins, allow the submitted prescriber_id (admin-assisted submissions)
if (userRole === "provider") {
  body.prescriber_id = user.id;
}
1c. Patient existence check (after creating supabaseAdmin):

// Verify patient exists
const { data: patientRecord } = await supabaseAdmin
  .from("patients")
  .select("id")
  .eq("id", body.patient_id)
  .single();
if (!patientRecord) {
  return NextResponse.json(
    { success: false, error: "Patient not found" },
    { status: 404 },
  );
}
1d. Provider-patient mapping check (after getting provider profile):

// For providers, verify they are mapped to this patient
if (userRole === "provider") {
  const { data: mapping } = await supabaseAdmin
    .from("provider_patient_mappings")
    .select("id")
    .eq("provider_id", provider.id)
    .eq("patient_id", patientRecord.id)
    .single();
  if (!mapping) {
    return NextResponse.json(
      { success: false, error: "You do not have access to this patient" },
      { status: 403 },
    );
  }
}
IMPORTANT: Variable naming
Use patientRecord for the existence check (selects only id)
Use a DIFFERENT variable name (e.g., patientDetails) for the later query that fetches data, phone, email, physical_address for the prescription payload
Do NOT reuse the same variable name — it causes TypeScript shadowing issues
FIX 2: app/api/prescriptions/[id]/submit-to-pharmacy/route.ts
What was wrong
Had a hardcoded fallback secret "webhook-auto-submit" — anyone who knew this string could bypass auth
No role check for authenticated (non-internal) users
When INTERNAL_API_SECRET was not set, accepted ANY request with an x-internal-secret header as trusted
What to change
2a. Replace the internal auth logic at the top of the POST handler:

const internalSecret = request.headers.get("x-internal-secret");
const configuredSecret = process.env.INTERNAL_API_SECRET;
let isInternalCall = false;
if (configuredSecret && configuredSecret.length > 0) {
  if (internalSecret === configuredSecret) {
    isInternalCall = true;
  } else if (internalSecret) {
    return NextResponse.json(
      { success: false, error: "Invalid internal secret" },
      { status: 403 },
    );
  }
} else if (internalSecret !== undefined && internalSecret !== null) {
  if (process.env.NODE_ENV === "production") {
    console.error("⚠️ [submit-to-pharmacy] INTERNAL_API_SECRET not configured in production — rejecting internal call. Set INTERNAL_API_SECRET.");
    // Falls through to session auth below
  } else {
    console.warn("⚠️ [submit-to-pharmacy] INTERNAL_API_SECRET not configured — accepting internal call in dev mode.");
    isInternalCall = true;
  }
}
2b. Add role check for non-internal callers:

if (!isInternalCall) {
  const { user, userRole } = await getUser();
  if (!user) {
    return NextResponse.json(
      { success: false, error: "Unauthorized" },
      { status: 401 },
    );
  }
  if (!userRole || (userRole !== "provider" && userRole !== "admin")) {
    return NextResponse.json(
      { success: false, error: "Only providers and admins can submit to pharmacy" },
      { status: 403 },
    );
  }
}
2c. REMOVE any hardcoded fallback like "webhook-auto-submit" — search the entire file and delete any reference to this string.

FIX 4: app/api/basic-emr/orders/review-list/route.ts
What was wrong
No provider-patient scope — providers could see ALL patients' orders
What to add
After verifying the user is a provider, scope the query to only their mapped patients:

let patientIds: string[] | null = null;
if (userRole.role === "provider") {
  const { data: providerData } = await supabase
    .from("providers")
    .select("id")
    .eq("user_id", user.id)
    .single();
  if (providerData) {
    const { data: mappings } = await supabase
      .from("provider_patient_mappings")
      .select("patient_id")
      .eq("provider_id", providerData.id);
    patientIds = (mappings || []).map((m: { patient_id: string }) => m.patient_id);
  } else {
    patientIds = []; // No provider record = no patients
  }
}
// Admins: patientIds stays null = see all orders
Then when building the orders query, filter by patient IDs:

if (patientIds !== null && patientIds.length > 0) {
  // Get user_ids for these patient_ids (orders use user_id, not patient_id)
  const { data: patientUserIds } = await supabase
    .from("patients")
    .select("user_id")
    .in("id", patientIds);
  const userIds = (patientUserIds || []).map((p: { user_id: string }) => p.user_id);
  if (userIds.length > 0) {
    ordersQuery = ordersQuery.in("user_id", userIds);
  } else {
    return NextResponse.json({ success: true, data: [] });
  }
} else if (patientIds !== null && patientIds.length === 0) {
  return NextResponse.json({ success: true, data: [] });
}

FIX 6: app/api/basic-emr/orders/link-appointment/route.ts
What was wrong
No role check
No provider-patient mapping check
What to add
6a. Role check at the top:

if (!userRole || (userRole !== "provider" && userRole !== "admin")) {
  return NextResponse.json(
    { error: "Only providers and admins can link appointments to orders" },
    { status: 403 },
  );
}
6b. Provider-patient mapping check — IMPORTANT: Place this AFTER the patient reconciliation block:

This route has a complex section where it reconciles mismatched patient_id between appointments and orders (creates patient records, updates appointment patient_id). The mapping check MUST go AFTER that reconciliation, when appointment.patient_id is final:

// Now check provider-patient mapping (after reconciliation so patient_id is final)
if (userRole === "provider" && appointment.patient_id) {
  const adminClient = createAdminClient();
  const { data: providerData } = await adminClient
    .from("providers")
    .select("id")
    .eq("user_id", user.id)
    .single();
  if (!providerData) {
    return NextResponse.json(
      { error: "Provider record not found" },
      { status: 403 },
    );
  }
  const { data: mapping } = await adminClient
    .from("provider_patient_mappings")
    .select("id")
    .eq("provider_id", providerData.id)
    .eq("patient_id", appointment.patient_id)
    .single();
  if (!mapping) {
    return NextResponse.json(
      { error: "You do not have access to this patient" },
      { status: 403 },
    );
  }
}
WHY after reconciliation: The reconciliation code may change appointment.patient_id — for example, creating a new patient record and updating the appointment. If you check the mapping BEFORE reconciliation, you'd be checking against the old (possibly wrong) patient_id, which would reject valid requests.

ENVIRONMENT VARIABLE REQUIRED
You MUST set INTERNAL_API_SECRET as an environment variable (both on Replit and on whatever hosting platform you use). Generate a strong random value:

INTERNAL_API_SECRET=<random-64-char-hex-string>
This is used for the mark-paid → submit-to-pharmacy internal server-to-server call. Without it:

In development: internal calls are accepted with a warning
In production: internal calls are REJECTED and fall through to session auth (which won't work for server-to-server calls)
