Subject: Bug — Profile Tab Save Wipes Physical Address

Hi Tomas,

I found a bug causing the "Missing: Physical Address" modal to appear on the Prescriptions page even after the provider has filled in their address.

What's happening:

The provider fills in their Physical Address on the Payment & Billing tab and saves it — the address goes into the database correctly. Later, when they go to the Profile tab and click "Save Changes," their address is erased from the database.

This happens because the Profile tab's save function (updatePersonalInfo in provider-profile-service.ts, line 106) includes physical_address in the update payload. Since the Profile tab doesn't actually show the address fields on screen, the form sends empty values, and the save overwrites the real address with null.

Then when the provider goes to create a prescription, the check on prescriptions/page.tsx (lines 526-531) looks up physical_address, finds it empty, and shows the "Complete Your Profile" modal.

Where the bug is:

File: features/provider-profile/services/provider-profile-service.ts
Line 106: physical_address: hasPhysicalAddressData ? data.physicalAddress : null
Line 107: billing_address: hasBillingAddressData ? data.billingAddress : null
The fix:

Remove physical_address and billing_address from the updateData object in the updatePersonalInfo function. These fields are only entered on the Payment & Billing tab, so they should only be saved by PaymentBillingForm.tsx — the Profile tab shouldn't touch them.

Also worth noting: the Profile tab saves empty address fields to localStorage (provider-profile-${userId}), which can make the problem worse when the form reloads and merges stale localStorage data over DB values. Consider excluding physicalAddress and billingAddress from the localStorage persistence in ProfileForm as well.

 Root Cause Found — Patient Address Missing from Pharmacy Payload

Hi Tomas,

We traced why the Feb 26 payload to Greenwich Rx had no patient address even though the patient's address was in the database.

Bug 1 — Patient Address Override (the main issue)

File: app/api/prescriptions/[id]/submit-to-pharmacy/route.ts, lines 188-190

const patientAddress = prescription.has_custom_address && prescription.custom_address
  ? prescription.custom_address
  : patient?.physical_address;
When has_custom_address is true but custom_address is an empty object {}, JavaScript treats {} as truthy. So the code picks the empty custom_address instead of the patient's real address from the database. All address fields become undefined and get silently dropped from the JSON payload.

This was added in commit bb4ee1a on Feb 10 ("Override custom address prescription"). Before that, the code read the patient address directly and it would have worked.

Fix: Check that custom_address actually has real data inside it before using it:

const hasValidCustomAddress = prescription.has_custom_address
  && prescription.custom_address
  && prescription.custom_address.street
  && prescription.custom_address.city
  && prescription.custom_address.state
  && (prescription.custom_address.zipCode || prescription.custom_address.zip);
const patientAddress = hasValidCustomAddress
  ? prescription.custom_address
  : patient?.physical_address;
Bug 2 — DoctorPhone always undefined

Same file, line 217: DoctorPhone: provider.phone

The providers table column is phone_number, not phone. So DoctorPhone is always undefined and never sent.

Fix: Change to provider.phone_number

Bug 3 — Patient Sex always "F"

Same file, line 200: Sex: prescription.patients.gender === "male" ? "M" : "F"

The patients table has no gender column. Gender is stored inside the data JSON field as patient.data.gender. So prescription.patients.gender is always undefined, the comparison is always false, and Sex always defaults to "F".

Fix: Change to prescription.patients.data?.gender === "male" ? "M" : "F"
