2. Webhook Signature Bypass (Authorize.Net)
In app/api/webhooks/authnet/route.ts line 19: the signature check only runs if the header exists. If someone sends a request without the x-anet-signature header, the webhook processes it without any verification. An attacker could fake payment confirmations, mark prescriptions as paid, and trigger pharmacy submissions.

3. Hardcoded Passwords in Source Code
force-seed-admins/route.ts has passwords in plain text: AIM2025! and Grin2025!. These are in the Git repo.

HIGH SEVERITY
4. Session Instability (the bug we discussed)
The Render logs confirmed sessions drop intermittently. Root cause: the middleware refreshes cookies via supabase.auth.getUser(), but if any response is returned before the cookie setAll completes, the browser and server go out of sync. The 2-minute cache TTL on role cookies is also very aggressive — it means roles are re-queried constantly, adding latency.

5. Hardcoded Credentials in invite-doctor Route
The invite-doctor route accepts a password in the request body and creates a user with that password. No strength validation, no rate limiting.

6. No Rate Limiting
No rate limiting on any endpoint — login, password reset, MFA verification, payment links. An attacker could brute-force MFA codes or spam login attempts.

MEDIUM SEVERITY
